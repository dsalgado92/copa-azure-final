### Por que a ordem importa (3.1 antes de 3.2)

A ADE-008 já verificou que os dois notifiers (`N8nWebhookNotifier`, `AlertWebhookNotifier`) são **fail-open/no-op** quando a URL está vazia — ou seja, tecnicamente seria possível derrubar a infra ANTES de limpar o código sem quebrar nada (o notifier só logaria "não configurado" em vez de tentar conectar). Mesmo assim, a ordem **código primeiro** é mais segura e didaticamente mais limpa: evita uma janela onde o código ainda "acredita" que pode existir um n8n, e mantém a regra geral do projeto de nunca deixar infra órfã referenciada por código morto.

### Troubleshooting esperado

| Sintoma | Causa | Mitigação |
|---|---|---|
| `az containerapp delete` falha com "resource in use" | Algum outro recurso (raro) referencia o Container App do n8n via VNet/DNS interno | Confirmar (Task 2.1) que nenhum outro serviço aponta para o FQDN antes de remover; se houver, resolver a dependência primeiro |
| Dump do PostgreSQL falha por falta de permissão | Firewall do Flexible Server bloqueia conexão externa | Usar Azure Cloud Shell (mesma VNet/rede) ou liberar temporariamente o IP do executor antes do dump; fechar o firewall de volta antes do drop |
| Cost Management ainda mostra linha residual do n8n dias depois | Latência normal de billing do Azure (24-48h) | Não é regressão — reconfirmar após esse prazo |
