**Conclusão original (constatação técnica, não uma decisão do `@sm`):** a ADE-007 endereçava a migração de usuários **existentes**; esta story precisava do caso **espelhado** (usuário **nato**-CIAM, sem histórico v1). - **(a) "É só a metade que faltava da Inv 6"** — o JIT é o mesmo padrão (vincular `oid` CIAM a uma linha `users`), só que o INSERT acontece on-demand em vez de em lote.

**A resposta foi a síntese das duas** (ver caixa de resolução acima): Inv 8, adicionada pela ADE-007 v1.2 e precisada pela v1.3.

## Story

**Como** desenvolvedor do squad do Living Lab (papel `@dev`, sob o gate `@architect`),
**eu quero** provisionar *just-in-time* um registro `users` (com `entra_oid` vinculado) no primeiro login/checkout de um cliente **nato-CIAM** (que nunca teve conta v1), expondo um endpoint `GET /api/v2/me` — restrito ao esquema **CIAM** (`CiamOnly`, nunca ao esquema Admin/workforce) — que resolve `X-Entra-OID` → `users.id` por precedência **oid → email-link → insert** (ADE-007 v1.2 Invariante 8),
**para que** um cliente que se cadastrou diretamente no Microsoft Entra External ID (sem nunca ter existido no v1) consiga completar uma compra v2 e se torne **cidadão de primeira classe** na base `users` (hoje invisível ao dashboard admin, que já lê `users`) — hoje `PurchaseRequest.UserId` (int, obrigatório) exige uma linha `users` pré-existente, gap **não** coberto pela migração batch da ADE-007 Inv 6 — e **sem que um token de admin possa jamais criar ou vincular essa linha em nome do operador** (Invariante 8.2).

2. **AC-2 — Endpoint `GET /api/v2/me`** (Azure Function HTTP trigger, atrás do gateway — herda ADE-004, mesmo padrão de `X-Entra-OID` já usado em `PurchaseEntryFunction.cs` linhas 126-134) implementa o contrato **resolve-or-provision** com precedência determinística (ADE-007 v1.2 Invariante 8): **(1) resolve por `oid`** — `SELECT id FROM dbo.users WHERE entra_oid = @oid`; se existe, retorna o `id` existente. **(2) resolve por `email` → LINK** — se não há linha por `oid` mas existe uma por `email` do claim (um usuário v1 não-migrado chegando via CIAM), vincula em vez de inserir: `UPDATE dbo.users SET entra_oid=@oid WHERE email=@email AND entra_oid IS NULL` — exatamente o UPDATE idempotente da migração de cadastro (Inv 6). **(3) provision** — se não há linha por `oid` nem por `email`, `INSERT` de nova linha `users` (nato-CIAM): `entra_oid=@oid`; `name`/`email` dos headers propagados pelo gateway (AC-6); `role` no `DEFAULT ('user')`; `password` = sentinela fail-closed (AC-5).
   *(Fonte: ADE-007 v1.2 Invariante 8 — contrato resolve-or-provision completo, 3 passos; padrão herdado `X-Entra-OID` de `PurchaseEntryFunction.cs`.)*

3. **AC-3 — Fence de esquema `CiamOnly`: `/api/v2/me` autorizado SOMENTE para o esquema CIAM, nunca para o Admin (workforce).** Verificado na fonte (`Gateway/Program.cs`): o transform que injeta `X-Entra-OID` é **global** (linhas 94-116 — roda para todos os clusters e qualquer usuário autenticado, seja `CiamScheme` ou `AdminScheme`, definidos nas linhas 236-237), e a `DefaultPolicy` das rotas v2 aceita **CIAM OU Admin** autenticado (linhas 393-396, `new AuthorizationPolicyBuilder(CiamScheme, AdminScheme).RequireAuthenticatedUser()`). **Sem este fence**, um token de **admin** (workforce) — que também carrega o claim `oid` — satisfaria `/api/v2/me`, o gateway injetaria o `oid` do **operador** como `X-Entra-OID`, e o resolve-or-provision (AC-2) criaria/vincularia uma linha de **CLIENTE** com a identidade do admin — corrompendo a base de clientes (e, pelo arm de LINK por email, potencialmente vinculando a linha de um cliente real ao operador). A nova policy `CiamOnly` é o **espelho exato** de `AdminOnlyPolicy` (já existente, linhas 388-389, 408-410 — `RequireRole(AdminRole)` sobre o esquema Admin), exigindo o esquema **`Ciam`** em vez do `Admin`. Um token workforce roteado pelo selector ao esquema `Admin` **não** satisfaz `CiamOnly` → **401/403** (forma exata — fixar o esquema vs. só checar autenticação — é decisão de `@dev`, mesma reflexão já registrada para `AdminOnlyPolicy`) — o resolve-or-provision **nunca** corre com um `oid` de admin.
   *(Fonte: ADE-007 v1.3 Invariante 8.2 — resolvida integralmente nesta ADE, nova nesta atualização da story; verificado `Gateway/Program.cs` linhas 94-116 (transform global de `X-Entra-OID`), 236-238 (`CiamScheme`/`AdminScheme`/`SelectorScheme`), 393-396 (`DefaultPolicy`), 388-389/408-410 (`AdminOnlyPolicy`, o espelho a seguir).)*

4. **AC-4 — Idempotência sob concorrência via os índices UNIQUE já existentes (`UQ_users_entra_oid` E `UQ_users_email`) + captura de violação, nunca SELECT-then-INSERT como única defesa:** chamadas repetidas ou concorrentes ao `/api/v2/me` para o **mesmo** `oid`/`email` NÃO criam duplicata — reusa `UQ_users_entra_oid` (índice UNIQUE filtrado, já criado por `phase-04-ciam-link.sql` na Story 2.11) **e** `UQ_users_email` (já existente, `schema.sql`) como guardas de concorrência, com captura de `SqlException` 2627/2601 → **re-resolve** (não apenas rejeita) — mesmo padrão de idempotência já usado em `PurchaseRepository`/`InsertOutcome.Duplicate`. Uma corrida entre dois primeiros-logins do mesmo `oid`/`email` colapsa em **uma** linha; o perdedor re-resolve e retorna o `id` vencedor.
   *(Fonte: verificado `phase-04-ciam-link.sql` linhas 55-67 (`UQ_users_entra_oid`); ADE-007 v1.2 Invariante 8 — "guardado pelos índices UNIQUE + captura 2627/2601 → re-resolve... espelha `PurchaseRepository.cs:111-119`"; padrão de idempotência herdado de `PurchaseRepository.cs`.)*

5. **AC-5 — `users.password NOT NULL` sem `DEFAULT` resolvido por SENTINELA fail-closed (RESOLVIDO por ADE-007 v1.2 Invariante 8.1 — não mais condicional):** verificado em `fifa2026-api/database/schema.sql` linha 20 (`password NVARCHAR(255) NOT NULL`). Um usuário nato-CIAM **não tem bcrypt** (a Microsoft gerencia a credencial — a lição didática da ADE-007 Inv 4: "a senha bcrypt não viaja"). A **abordagem** já está decidida: o INSERT grava um **valor sentinela nomeado, documentado e comprovadamente não-verificável** (não é um hash bcrypt válido) — o caminho de login v1 (`bcryptjs.compare`) **nunca** autentica esse usuário, fail-closed por construção. **Rejeitado tornar `password` nullable** (seria `ALTER COLUMN`, não-aditivo, violaria ADE-000 Inv 2). O que permanece delegado (não a abordagem — só o literal): o **valor exato** da sentinela + o teste que prova `bcryptjs.compare(qualquer_entrada, SENTINELA) === false` → **`@data-engineer`/`@dev`** (matriz de autoridade, ADE-007 v1.2).
   *(Fonte: verificado `fifa2026-api/database/schema.sql` linhas 16-29 (`CREATE TABLE users`); ADE-007 v1.2 Invariante 8.1 — decisão completa da abordagem, só o literal delegado.)*

6. **AC-6 — Demais campos NOT NULL do INSERT resolvidos via headers propagados pelo gateway (`X-Entra-Email`/`X-Entra-Name`, ADITIVOS — mecanismo precisado pela ADE-007 v1.2), não lidos de um token que a Function nunca valida:** como a Function **não** valida o JWT (Inv 7 — só o gateway o faz), ela não pode ler `email`/`name` diretamente de um token; o gateway passa a propagar esses claims como **headers adicionais** (`X-Entra-Email`, `X-Entra-Name` — extensão aditiva dos transforms já existentes, mesmo padrão de `X-Entra-OID`; ADE-004 inalterada no modelo de guardião). `name` (`NVARCHAR(100) NOT NULL`) ← `X-Entra-Name`; `email` (`NVARCHAR(255) NOT NULL UNIQUE`) ← `X-Entra-Email`. `role` já tem `DEFAULT ('user')` — sem ação. **A confirmar com o owner** (mesma cautela M-1/M-2 do gate da Story 2.11): disponibilidade real de `email`/`name` no user flow do tenant CIAM **e** que o email seja **verificado** (OTP/Google comprovam posse) — sem isso, o arm de LINK por email (AC-2, passo 2) poderia vincular-se a uma vítima v1 por um email não comprovado (flag de segurança (b) da ADE-007 Invariante 8).
   *(Fonte: ADE-007 v1.2 Invariante 8 — "Claims `name`/`email` — o gateway propaga aditivamente... `X-Entra-Email`/`X-Entra-Name`"; ADE-007 Invariante 8 flag de segurança (b); verificado `schema.sql` linhas 16-29.)*

## Dev Notes

### O gap, verificado linha a linha (Art. IV)

```csharp
// src/Fifa2026.V2.Functions/Models/PurchaseRequest.cs, linha 25-27
[JsonPropertyName("userId")]
[Range(1, int.MaxValue, ErrorMessage = "userId deve ser um inteiro positivo.")]
public int UserId { get; set; }
```

Um `int` obrigatório, `Range(1, ...)` — não há sentinela válido para "sem usuário v1". Um cliente nato-CIAM não tem de onde tirar esse valor hoje.

```sql
-- fifa2026-api/database/schema.sql, linhas 16-29
CREATE TABLE users (
    id            INT IDENTITY(1,1) NOT NULL,
    name          NVARCHAR(100) NOT NULL,
    email         NVARCHAR(255) NOT NULL,
    password      NVARCHAR(255) NOT NULL,   -- ⚠️ NOT NULL, sem DEFAULT — o achado central desta story
    role          NVARCHAR(20)  NULL CONSTRAINT DF_users_role DEFAULT ('user'),
    ...
    CONSTRAINT UQ_users_email UNIQUE (email)
);
```

### Groundwork já existente (reusado, não recriado)

```sql
-- fifa2026-api/database/migrations/phase-04-ciam-link.sql (Story 2.11, já aplicada)
ALTER TABLE dbo.users ADD entra_oid UNIQUEIDENTIFIER NULL;
CREATE UNIQUE INDEX UQ_users_entra_oid ON dbo.users(entra_oid) WHERE entra_oid IS NOT NULL;
```

Este índice UNIQUE filtrado é exatamente uma das duas guardas de idempotência que o JIT insert precisa (AC-4, junto com `UQ_users_email` já existente) — nenhuma nova migration de índice é necessária. Só o **preenchimento** via JIT (em vez de via UPDATE hands-on) é novo.

### Padrão herdado para ler `X-Entra-OID`

```csharp
// src/Fifa2026.V2.Functions/Functions/PurchaseEntryFunction.cs, linhas 126-134
private const string EntraOidHeader = "X-Entra-OID";
...
Guid? entraOid = null;
var entraOidHeader = req.Headers[EntraOidHeader].ToString();
if (!string.IsNullOrWhiteSpace(entraOidHeader) && Guid.TryParse(entraOidHeader, out var parsedOid))
{
    entraOid = parsedOid;
}
```

O endpoint `/api/v2/me` deve seguir **exatamente** este padrão (não inventar uma nova forma de ler a identidade) — a Function confia no header porque o gateway é o guardião único (ADE-004 Inv 1/5), nunca valida o token ela mesma.

### A Invariante 8 da ADE-007 (v1.2/v1.3) — a resposta ao gap mapeado no §Gate de design

A tabela no topo deste documento (§Gate de design) mapeia as 7 invariantes **originais** (v1.1) e mostra que nenhuma cobria 100% o caso nato-CIAM — Inv 6 era a mais próxima (migração v1→CIAM), mas no sentido inverso (usuário v1 existente → vínculo CIAM).
