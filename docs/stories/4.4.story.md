### Bloco B — Cache-pós-auth (ADE-009 §Consequences — P1 de segurança)

2. **AC-2 — Pipeline do gateway reordenado: `XCacheMiddleware` passa a rodar DEPOIS de `UseAuthentication`/`UseAuthorization`.** Ordem atual verificada (`Gateway/Program.cs` linhas 419-424): `UseCors`(1) → `UseRateLimiter`(2) → `UseMiddleware<XCacheMiddleware>`(3) → `UseAuthentication`(4) → `UseAuthorization`(5) → `MapReverseProxy`(6). Nova ordem: `UseCors`(1) → `UseRateLimiter`(2) → `UseAuthentication`(3) → `UseAuthorization`(4) → `UseMiddleware<XCacheMiddleware>`(5) → `MapReverseProxy`(6). **Efeito de segurança:** hoje, um cache **HIT** faz `XCacheMiddleware` responder e retornar **sem nunca chamar `_next()`** (verificado `XCacheMiddleware.cs` linhas 63-74, `return` após escrever a resposta cacheada) — ou seja, um HIT **nunca passa** por `UseAuthentication`/`UseAuthorization`, servindo o status de uma compra por até 30s **sem exigir token válido**, mesmo a rota `MapReverseProxy()` exigindo `.RequireAuthorization()` (linha 446) para todo o resto. Com o reorder, TODA request (HIT ou MISS) passa primeiro por auth — só chega ao cache se autenticada.
   *(Fonte: verificado `Gateway/Program.cs` linhas 419-424 (pipeline atual); `XCacheMiddleware.cs` linhas 37-74 (short-circuit em HIT); linha 446 (`RequireAuthorization()` do proxy); ADE-009 §Consequences — "HIT serve status sem token por 30s".)*

3. **AC-3 — O comentário de cabeçalho de `XCacheMiddleware.cs` (linhas 14-23) é atualizado para refletir a nova posição e o novo racional.** O texto atual afirma explicitamente *"Mantém a posição de pipeline do design original (antes da autenticação): o status é keyed pelo correlationId e não depende do usuário"* — essa frase fica **falsa** após o reorder e PRECISA ser substituída (não só o código; Art. IV — a doc não pode contradizer o comportamento real). Novo racional: o cache continua keyed por path/correlationId (não por usuário — a resposta em si não muda por quem pergunta), mas agora só é alcançado **depois** que a autenticação/autorização já validaram a request — o cache economiza a chamada ao backend, não a checagem de identidade.
   *(Fonte: verificado `XCacheMiddleware.cs` linhas 14-23 — texto exato citado.)*

4. **AC-4 — Smoke: "cache HIT só com auth".** `GET /purchase/{correlationId}` **sem** `Authorization` header → **401** tanto na 1ª chamada (MISS, nunca chega ao cache) quanto numa eventual 2ª chamada dentro da janela de 30s (que HOJE seria HIT sem token — após esta story, continua 401, pois a auth roda antes do cache). `GET` com token válido → 200 na 1ª (MISS, `X-Cache: MISS`) e 200 na 2ª dentro de 30s (HIT, `X-Cache: HIT`, sem nova chamada ao backend). Este é o smoke central desta AC (ADE-009 §Validation — não item formal, mas item de verificação do plano §10 "cache HIT só com auth").
   *(Fonte: verificado `XCacheMiddleware.cs` linhas 40-58 (rota cacheável = `purchase-get`, keyed por path+querystring); ADE-009 §Consequences; plano §10.)*

5. **AC-5 — Regra de elegibilidade de cache (rota `purchase-get`, GET-only) não muda** — só a POSIÇÃO no pipeline muda, não a lógica de decisão de cache (`CacheableRouteId`, `TimeSpan.FromSeconds(30)`, só respostas 200 armazenadas — verificado linhas 30-31, 40, 94-97). Nenhuma rota nova passa a ser cacheável nesta story.
   *(Fonte: verificado `XCacheMiddleware.cs` linhas 25-31, 40, 94-97 — código de decisão intocado.)*

### Bloco C — `UseForwardedHeaders` (ADE-009 §Consequences — rate-limit)

6. **AC-6 — `app.UseForwardedHeaders(...)` adicionado ao pipeline do gateway**, posicionado **antes** de `UseRateLimiter` (o mais cedo possível — precisa reescrever `HttpContext.Connection.RemoteIpAddress` antes de qualquer middleware que o leia). Hoje, o particionador do rate-limiter (`Gateway/Program.cs` linhas 169-173, `httpContext.Connection.RemoteIpAddress`) vê o IP do **ingress** do Container Apps Environment (sempre o mesmo, ou um pool pequeno) — não o IP real do cliente — tornando a partição "por IP" **efetivamente global** (todo mundo compete pelo mesmo bucket de 5/min em vez de ter seu próprio limite). `builder.Services.Configure<ForwardedHeadersOptions>(...)` define `ForwardedHeaders = ForwardedHeaders.XForwardedFor | ForwardedHeaders.XForwardedProto`.
   *(Fonte: verificado `Gateway/Program.cs` linhas 162-182 (rate limiter usa `RemoteIpAddress` como chave de partição); ADE-009 §Consequences — "rate-limit sem `UseForwardedHeaders`"; plano §5 item 12.)*

7. **AC-7 — `KnownNetworks`/`KnownProxies` configurados de forma consciente (gotcha real do ASP.NET Core):** o middleware `ForwardedHeadersMiddleware` só confia em `X-Forwarded-For` de proxies em `KnownNetworks`/`KnownProxies` — por padrão, restrito à rede loopback, o que **não bate** com o Azure Container Apps ingress (não é loopback). A configuração precisa **limpar** essas listas (`options.KnownNetworks.Clear(); options.KnownProxies.Clear();`) para aceitar o header do ingress do CAE — decisão documentada e assumida (o CAE já é a borda confiável do ambiente; não há um proxy adicional não-confiável entre o ingress e o gateway). `@dev` documenta essa decisão em comentário no código (mesmo padrão de honestidade dos outros comentários do `Program.cs`).
   *(Fonte: comportamento documentado do ASP.NET Core `ForwardedHeadersMiddleware` — não inventado; aplicação ao contexto de Azure Container Apps é decisão desta story, registrada como tal.)*

8. **AC-8 — Smoke: rate-limit passa a particionar por IP real (ou, no mínimo, deixa de colapsar todo o tráfego num único bucket).** Duas origens de IP distintas (simuladas via `X-Forwarded-For` diferente em teste) não compartilham mais o mesmo contador de 5/min.
   *(Fonte: ADE-009 §Consequences; verificado `Gateway/Program.cs` linhas 167-181 — lógica de partição.)*
