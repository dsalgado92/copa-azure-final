## Story

**Como** aluno da Grande Final (que já completou Oitavas e Quartas — tem sua própria infraestrutura Azure e identidade CIAM/admin configuradas),
**eu quero** clicar "Use this template" para criar meu **próprio** repositório (não um fork) a partir de `fifa2026-final-lab`, editar **uma única linha** (`sufixo`) em `lab.config.json`, configurar **um único secret** (`AZURE_CREDENTIALS`) e rodar `acao=check` para confirmar que meu ambiente está pronto antes de `acao=tudo`,
**para que** eu não precise preencher as **~15 variáveis + 5 secrets** que as Quartas exigiam nem herde a classe de bugs de merge de fork que atrapalhou turmas anteriores — e possa confiar, com um pré-flight automatizado, que meu ambiente está correto antes de começar os labs de verdade (Blindar/F5/F6/Diploma).

## `[AUTO-DECISION]` — a pendência mais delicada da simplificação, resolvida aqui

### Vars de identidade (CIAM/Admin) e Secrets de deploy num repositório NOVO → **consultados dinamicamente, não re-inseridos**

**O problema:** `VITE_CIAM_AUTHORITY`/`CLIENT_ID`, `VITE_ADMIN_TENANT_ID`/`CLIENT_ID` são variáveis de **build-time do frontend** (Vite as embute no bundle estático — não há "runtime" de KV reference para uma SPA já compilada). `SQL_CONNECTION_STRING`, `AZURE_FRONTEND_PUBLISH_PROFILE`, `AZURE_BACKEND_PUBLISH_PROFILE` são específicas do aluno. Como um repositório **novo** (Template Repo, não uma continuação do fork das Quartas) resolve esses valores sem pedir ao aluno para digitá-los de novo?

**Decisão:** o workflow **consulta dinamicamente** os recursos Azure **já existentes** do aluno (criados nas Oitavas/Quartas, nomeados pela **mesma convenção** de `sufixo` — AC-4) usando o `AZURE_CREDENTIALS` da turma:
- `SQL_CONNECTION_STRING` **deixa de existir** — a Story 4.1 já move as conexões de aplicação para MI-AAD; não há connection string com senha para propagar.
- `AZURE_FRONTEND_PUBLISH_PROFILE`/`AZURE_BACKEND_PUBLISH_PROFILE` são substituídos por `az webapp deploy`/`az functionapp deployment` autenticados via `AZURE_CREDENTIALS` (Service Principal) — **nenhum publish profile** precisa ser baixado/colado pelo aluno.
- `VITE_CIAM_*`/`VITE_ADMIN_*` (build-time do frontend): o workflow **lê os App Settings já configurados** no frontend App Service existente do aluno (`az webapp config appsettings list`, ou lê os equivalentes já persistidos nas GitHub Variables/Secrets do FORK original das Quartas, se o `@devops` decidir reusar aquele repositório como fonte — decisão de implementação) e os injeta como `VITE_*` no build **sem** o aluno redigitar. Se o Diploma/chatbot não exigirem rebuild do frontend inteiro (só rotas novas adicionadas), este passo pode nem ser necessário — **decisão final de arquitetura de build cabe a `@dev`/`@devops` na implementação**, registrada aqui como direção, não como literal.

**Por quê (justificativa):**
- **Honra a "URL/valor nunca hardcoded" (ADE-003) e "zero segredo no GitHub" (ADE-009 Inv 2) simultaneamente** — em vez de inventar um mecanismo novo de propagação de segredo, reusa a MESMA fonte da verdade (os recursos Azure reais do próprio aluno) que já existe.
- **Evita reintroduzir GitHub Variables pela porta dos fundos** — se o aluno tivesse que re-digitar os 5-6 valores de identidade no repositório novo, a simplificação "15→1" seria só nominal.
- **Alternativa registrada (não escolhida):** pedir ao aluno para copiar manualmente os valores do fork antigo para o repo novo — rejeitada por reintroduzir exatamente a fricção que esta story existe para eliminar.
